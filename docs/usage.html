prerequisites

You need access to the Tufts HPC cluster before using tccp. Request access here if you haven't already. The Tufts HPC documentation is a good reference for the cluster itself.

getting started

You need to be on the Tufts network (campus WiFi or Cisco AnyConnect VPN). On campus WiFi, you'll need to approve a Duo push every time you connect. The VPN avoids this. 

# save credentials (one time)
$ tccp setup

# set up your project
$ cd my-ml-project
$ tccp register

# connect to the cluster
$ tccp

tccp setup saves your Tufts username and password. tccp register walks you through creating a tccp.yaml config file. Then tccp connects you to the cluster REPL.

tccp.yaml

Lives in your project root. tccp register creates it for you, or write it by hand. Everything is optional — see settings for the full reference. 

type: python-pytorch
gpu: a100
env: .env
output: output
cache: cache

jobs:
  train:
    script: train.py
    args: "--lr 0.001"
    time: "4:00:00"
    ports: [6006]

Put API keys in a .env file — tccp syncs it automatically and it's never tracked by git.

running jobs

Type run train and tccp handles the rest: finds or creates a SLURM allocation, sets up the environment, syncs your code, and launches. This happens in the background so you can keep using the REPL. You can pass extra arguments inline or run multiple jobs at once. 

tccp> run train --epochs 100 --batch-size 64
tccp> run eval

viewing output

When a job is running, view attaches you to the live output stream. Ctrl+C cancels the job, Ctrl+\ detaches (job keeps running — reattach anytime with view again). Ctrl+V opens the full output buffer in your text editor for searching and scrolling. You can also use the output command to do the same thing from the REPL.

interactive shell

open drops you into a bash shell on the compute node, inside the same container as your job. Your code, venv, and env vars are all there. 

tccp> open train
train@c0178> python -c "import torch; print(torch.cuda.is_available())"
True

disconnects and output

Jobs survive disconnects. If your laptop sleeps or your WiFi drops, the job keeps running. Reconnect with tccp and everything is still there. When a job finishes, output is downloaded to your machine automatically — if you're offline, it downloads next time you connect.

allocations

tccp reuses SLURM allocations across jobs. If you run a second job that needs the same resources, it skips the queue and launches instantly. When you exit tccp, idle allocations are released automatically. You can also manage them manually with allocs and dealloc.

port forwarding

Declare ports in your config and they're forwarded to localhost automatically. No SSH tunnel setup — just open http://localhost:6006. 

jobs:
  train:
    script: train.py
    ports: [6006]

file sync

tccp syncs your project files to the compute node before each run.

  • Respects .gitignore (or .tccpignore). Auto-excludes .git/, __pycache__/, .venv/, node_modules/.
  • Only changed files are sent (incremental sync by content hash).
  • The env file is always synced, even if gitignored.
  • requirements.txt is auto-installed and cached by hash.
  • Remote directory structure mirrors local exactly: 
    local ./data/train.csv  →  remote <scratch>/data/train.csv
local ./train.py        →  remote <scratch>/train.py

project structure

A typical tccp project looks like:

my-project/
  tccp.yaml              # project config
  requirements.txt       # auto-installed into container venv
  .env                   # API keys, secrets (auto-synced, gitignored)
  .gitignore
  train.py               # your scripts
  eval.py
  data/                  # rodata — synced once per allocation
    train.csv
    val.csv
  output/                # downloaded from cluster when job finishes

expected timing

When you run <job>, initialization happens in the background. The first run for a new project is the slowest because the container image and environment must be set up. After that, everything is cached and re-runs are fast.

StageFirst runRe-runs
SLURM allocation10–120s0s (reused)
Container pull5–15 min0s (cached)
Venv creation20–30s0s (cached)
Tool install (uv, dtach)5–15s0s (cached)
File sync2–30s (full)1–3s (incremental)
Requirements (uv)5–30s0s (cached by hash)
Job launch1–2s1–2s
Total~1–20 min3–10s

Use initlogs <job> to see real-time progress of each stage.

good to know

  • requirements.txt — put it in the project root. For python-pytorch, don't list torch, torchvision, or torchaudio — they're pre-installed in the container.
  • Output — write files to your output directory. tccp downloads it automatically when the job finishes.
  • Cache — use the cache directory for model weights, datasets, or anything you don't want to re-download. Point HF_HOME, TRANSFORMERS_CACHE, etc. into it.
  • Time limits — set realistic time: values. Allocations hold the node for the full duration, burning SUs even if the job finishes early. Use dealloc to release idle ones.
  • Refresh — if you edit tccp.yaml, type refresh to pick up changes without reconnecting.
