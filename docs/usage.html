prerequisites

You need access to the Tufts HPC cluster before using tccp. Request access here if you haven't already. The Tufts HPC documentation is a good reference for the cluster itself.

getting started

You need to be on the Tufts network (campus WiFi or Cisco AnyConnect VPN). On campus WiFi, you'll need to approve a Duo push when you connect. The VPN avoids this. 

# save credentials (one time)
$ tccp setup

# set up your project
$ cd my-ml-project
# create tccp.yaml (see settings page for reference)

# start a session
$ tccp start

# attach to the persistent shell
$ tccp shell

tccp setup saves your Tufts username and password. Then create a tccp.yaml in your project root (the only required field is container). tccp start allocates a compute node, pulls your container, syncs your files, and starts a persistent shell. tccp shell attaches you to it.

tccp.yaml

Lives in your project root. The only required field is container. See settings for the full reference. 

container: pytorch/pytorch:2.6.0-cuda12.4-cudnn9-runtime
init: "pip install -r requirements.txt"
gpu: a100
time: 4h
output: output/
ports: [6006]

the workflow

tccp gives you a persistent shell on a compute node inside your container. The basic loop is:

  1. tccp start — allocate a node and set everything up
  2. tccp shell — get a shell, run your code interactively
  3. Edit code locally, press Ctrl+S to sync changes and reattach
  4. tccp stop when done — pulls output and releases the node

shell controls

Inside tccp shell, you're in a normal bash session on the compute node, inside your Singularity container with GPU access.

keyaction
Ctrl+SDetach from shell, sync code changes from laptop, reattach. This is the fast edit-sync-test loop.
Ctrl+DExit shell normally. The dtach session ends. Job allocation stays alive for reuse.

If your laptop sleeps or WiFi drops, the shell session survives. Just run tccp shell again to reattach.

port forwarding

Declare ports in your config and they're forwarded to localhost automatically during tccp shell. No SSH tunnel setup needed. 

ports: [6006, 8888]

Then open http://localhost:6006 for TensorBoard, http://localhost:8888 for Jupyter, etc.

file sync

tccp syncs your project files to the compute node on tccp start and whenever you press Ctrl+S or run tccp sync.

  • Respects .tccpignore (or .gitignore). Auto-excludes .git/, __pycache__/, .venv/, node_modules/, build/, output/, and others.
  • Only changed files are sent (incremental sync by mtime + size).
  • Remote directory structure mirrors local exactly.
  • Files created on the compute node that aren't in your local project are not deleted by sync.

output

The output/ directory (or whatever you set in config) is bind-mounted from NFS into your container. Files written there persist immediately and are pulled back to your laptop on tccp sync or tccp stop.

project structure

A typical tccp project:

my-project/
  tccp.yaml              # project config
  requirements.txt       # installed by init command
  .gitignore
  train.py               # your scripts
  eval.py
  data/                  # rodata — bind-mounted from NFS
    train.csv
  output/                # synced back from cluster

The first run is slow because the container must be pulled. After that, it's cached (on NFS if cache-containers: true, or on compute /tmp by default). Subsequent tccp start calls mainly wait for SLURM allocation.

good to know

  • container is required — any Docker image works. It's pulled as a Singularity SIF file.
  • init runs once per start — use it for pip install -r requirements.txt or other setup. If no init: is set but tccp_init.sh exists, that runs instead.
  • output/ is on NFS — files written to output/ inside the container persist immediately.
  • rodata for large read-only data — bind-mounted from NFS home, not re-synced every time.
  • time format — accepts 4h, 30m, 1d, or HH:MM:SS.
  • GPU auto-selection — if no gpu is set in config, tccp picks the best available GPU by VRAM on the partition.
